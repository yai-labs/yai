# D18.3 Program Lifecycle Grammar

## Canonical Status Set
- `draft`: proposed, not yet ratified.
- `accepted`: ratified decision, pending broad operational activation or staged rollout.
- `active`: currently authoritative and actively enforced.
- `superseded`: replaced by a newer decision; retained for lineage.
- `historical`: no longer authoritative and preserved for traceability only.

## Allowed Status by Family
- ADR: `draft|accepted|active|superseded|historical`
- RFC: `draft|accepted|active|superseded|historical`
- Milestone Packs (live index): `active|historical`
- Program Reports: `active|superseded|historical`

## Required Lifecycle Markers (Program Core)
- `role`
- `status`
- `decision_id`
- `owner_domain`
- `supersedes`
- `superseded_by`
- `implements` (when decision maps upstream/downstream)
- `evidenced_by` (when operational evidence applies)
- `related` (non-normative links)

## Transition Rules
- `draft -> accepted` when ratified by governance review.
- `accepted -> active` when adopted as current runtime/program authority.
- `active -> superseded` only with explicit replacement and backlink.
- `superseded -> historical` only when fully retired from active references.

## Supersession Contract
- If `status` is `superseded` or `historical`, `superseded_by` must be non-empty.
- If `supersedes` is declared, referenced docs should list reciprocal `superseded_by` unless historical exception is documented in ledger.

## Modeling Rules
- `implements` links design intent to concrete ADR/MP/report execution surfaces.
- `evidenced_by` links decisions to reports/runbooks proving runtime behavior.
- Reports are evidence surfaces and cannot supersede architecture authority.

## Validation Rules
- One active/accepted owner per `decision_id`.
- No unknown statuses.
- No historical doc listed as active in `DECISION_LEDGER`.
- Ledger rows must reference existing files and matching lifecycle metadata.
