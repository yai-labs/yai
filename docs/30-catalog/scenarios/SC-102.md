# SC-102 - Core Qualification: Contain -> Evidence (No Mind) - Cross-Domain

## 1) Purpose
Qualificare il Core deterministico dimostrando che la grammatica runtime:Ã¹

`trigger -> contesto -> authority/contract -> decision -> enforcement -> evidence`

resta identica attraversando i 9 D-Major, cambiando solo la semantica del Domain Pack (non il runtime).

SC-102 e il "Gate A" della credibilita v0.1: prova che l'infrastruttura decide, blocca e produce prove anche senza Mind e senza Cockpit.

## 2) Scenario model (parametrico)
SC-102 e uno scenario spec parametrico:

- Domain Pack: `Dk` (uno tra D1..D9)
- Workload: un servizio reale nel workspace (o reference workload)
- Forbidden Effect: un'azione esterna non autorizzata definita dal Domain Pack
- Baseline/Contracts: regole versionate che determinano `allow|deny|quarantine`

In pratica: stessa forma, domini diversi.

## 3) Preconditions
- Runtime up (functional pipeline): boot -> root -> kernel -> engine, where Engine is a shared runtime plane and workspace context is dispatched via metadata, not process topology (ADR-009).
- Workspace isolation attivo e `ws_id` non falsificabile.
- Control plane: handshake + contesto (session, role, arming).
- Engine readiness e validata via end-to-end RPC probe (`yai engine --ws <id> ...`) mediata da Root/Kernel; `engine` control socket puo essere non esposto in root/kernel-mediated mode.
- Baseline contratti/policy caricata e hashabile.
- Decision vocabulary stabile: `allow | deny | quarantine | rate_limit`.
- Evidence pipeline attiva (trace stream + materializzazione persistente).

## 4) Trigger (stimolo)
Il trigger e sempre: "un workload nel workspace tenta un Forbidden Effect".

Esempi per D-Major (semantica diversa, grammatica identica):
- D1 Digitale: egress verso destinazione non contrattualizzata / provider call fuori registry.
- D2 Fisico: comando attuatore fuori envelope (es. set valve open senza authority) mediato via adapter.
- D3 Biologico: export/uso campione fuori consenso o fuori catena di custodia (richiesta di accesso dati campione).
- D4 Sociale-Istituzionale: atto/procedura senza step autorizzativo (es. approva pratica fuori ruolo).
- D5 Economico: autorizzazione pagamento/credito fuori policy o fuori limite contrattuale.
- D6 Operativo: reroute/logistics action fuori finestra o senza escalation corretta.
- D7 Cognitivo-Culturale: pubblicazione/ranking/enforcement senza provenance o fuori policy.
- D8 Scientifico: pipeline run con parametri non riproducibili / esclusioni non dichiarate.
- D9 Ambientale/Climatologico: telemetry ingest o publish di metriche senza calibrazione/provenance/params lock.

Nota: il runtime vede sempre effetto richiesto, non la filosofia del dominio.

## 5) Expected Core Behavior (Grammar)
Per ogni Domain Pack:

1. Context
- `ws_id`, `trace_id`, principal/role, baseline hash, arming state.

2. Authority/Contract
- lookup contratti/policy del Domain Pack (versionati).

3. Decision
- outcome + reason code stabile (es. `DENY / QUARANTINE`).

4. Enforcement
- effetto esterno non avviene (fail-closed).
- metrica hard: nessun effetto irreversibile nel perimetro del pack.

5. Evidence
- record strutturato + baseline hash + riferimento contratto + outcome + reason.
- indicizzazione per replay/forensics.

## 6) Claims (cosa dimostra)
C1. Fail-closed: se non autorizzato, l'effetto non avviene.  
C2. Determinismo decisionale: outcome + reason code sono stabili a parita di baseline e input.  
C3. Evidence-first: ogni decisione produce prova strutturata con baseline hash.  
C4. Workspace scoping: evidenze non ambigue (`ws_id` non falsificabile).  
C5. Cross-domain coherence: cambiando Domain Pack, la grammatica resta invariata.  
C6. Minimal disclosure: l'evidence contiene il necessario (non leakage inutile).  

## 7) Non-goals
- Mind/LLM reasoning, remediation automatica, patch generation.
- UI/cockpit.
- Orchestrazione enterprise completa (ticketing/SOAR replacement).

## 8) KPIs
- time-to-contain (p50/p95)
- forbidden effect success rate (target: 0)
- evidence completeness (target: 100% campi required)
- repeatability (3/3 run coerenti)
- cross-domain coverage (quanti D-Major coperti con pack validi)

## 9) Acceptance (PASS/FAIL)
PASS se:
- effetto non autorizzato bloccato (fail-closed)
- decision record: outcome + reason + baseline hash
- evidence pack completo e indicizzato
- 3 run coerenti per lo stesso Domain Pack e baseline

FAIL se:
- l'effetto avviene anche una sola volta
- evidence/decision ambigue o mancanti
- check required in SKIP
